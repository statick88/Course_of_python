# Guía del Instructor — Módulo 10: Proyecto Final

## Objetivos de Aprendizaje

Al finalizar este módulo, el estudiante será capaz de:

1. Planificar un proyecto desde cero
2. Estructurar un proyecto en módulos
3. Integrar todos los conceptos del curso
4. Presentar y defender su código
5. Aplicar todas las buenas prácticas aprendidas

## Duración Estimada

- **Planificación**: 2 horas
- **Desarrollo**: 8 horas (trabajo autónomo)
- **Presentación**: 2 horas
- **Total**: 12 horas (distribuidas en 2 semanas)

## Estructura de la Sesión

### Sesión 1: Planificación (2 horas)

| Tiempo | Actividad | Material |
|--------|-----------|----------|
| 0-30 min | Presentación proyectos | 01-planificar-proyecto.qmd |
| 30-75 min | Estructura de proyecto | 02-estructura-proyecto.qmd |
| 75-120 min | Elegir proyecto + planificar | Formar equipos |

### Sesión 2-4: Desarrollo (trabajo autónomo)

- Revisión de avance (30 min por estudiante/equipo)
- Consultas individuales
- Código review progresivo

### Sesión 5: Presentaciones (2 horas)

| Tiempo | Actividad |
|--------|-----------|
| 0-60 min | Presentaciones (10 min cada uno) |
| 60-120 min | Feedback + evaluación |

## Proyectos Disponibles

### Proyecto 1: Gestor de Tareas
- **Complejidad**: Media
- **Conceptos**: Listas, diccionarios, archivos JSON, funciones
- **Duración**: 6-8 horas

### Proyecto 2: Tienda de Productos
- **Complejidad**: Media
- **Conceptos**: Diccionarios anidados, CSV, estadísticas
- **Duración**: 6-8 horas

### Proyecto 3: Analizador de Datos
- **Complejidad**: Alta
- **Conceptos**: CSV,统计局, funciones, modularización
- **Duración**: 8-10 horas

## Criterios de Evaluación del Proyecto

| Criterio | Peso | Descripción |
|----------|------|-------------|
| **Funcionalidad** | 30% | El programa funciona, cumple requisitos |
| **Estructura** | 20% | Módulos bien separados, responsabilidades claras |
| **Calidad de código** | 15% | PEP 8, nombres descriptivos, funciones cortas |
| **Documentación** | 15% | README completo, docstrings |
| **Manejo de errores** | 10% | Validación, try/except apropiados |
| **Presentación** | 10% | Explicación clara, responde preguntas |

## Rúbrica de Evaluación

### Funcionalidad (30%)

| Nivel | Puntuación |
|-------|------------|
| Excelente (27-30) | Todas las funcionalidades + extras funcionan |
| Bueno (22-26) | Funcionalidades principales funcionan |
| Satisfactorio (18-21) | Funcionalidad básica funciona |
| Necesita mejorar (0-17) | No funciona o incompleto |

### Estructura (20%)

| Nivel | Puntuación |
|-------|------------|
| Excelente (18-20) | Módulos claros, separación completa |
| Bueno (14-17) | Mayoría modularizado |
| Satisfactorio (10-13) | Alguna estructura |
| Necesita mejorar (0-9) | Todo en un archivo |

### Presentación (10%)

| Nivel | Puntuación |
|-------|------------|
| Excelente (9-10) | Explica código, justifica decisiones, responde bien |
| Bueno (7-8) | Explica lo básico, algunas preguntas |
| Satisfactorio (5-6) | Presentación limitada |
| Necesita mejorar (0-4) | No puede explicar su código |

## Preguntas para la Presentación

El instructor debe hacer estas preguntas durante la presentación:

1. "¿Cómo organizaste el código?"
2. "¿Qué decisiones de diseño tomaste y por qué?"
3. "¿Cómo manejas los errores?"
4. "¿Qué mejorarías si tuvieras más tiempo?"
5. "¿Qué aprendiste durante este proyecto?"

## Solución de Problemas Comunes

### El estudiante no sabe por dónde empezar
**Ayuda:** "Primero define qué datos necesitas guardar. Luego piensa qué funciones necesitas. Empieza por lo más simple."

### Código muy largo en una función
**Ayuda:** "Si tu función tiene más de 30 líneas, probablemente hace varias cosas. Divídela."

### No quiere usar módulos separados
**Ayuda:** "Imagina que tienes que cambiar algo después de 3 meses. ¿Qué sería más fácil de encontrar?"

### El proyecto se queda muy básico
**Ayuda:** "Agrega estadísticas, filtros, o alguna funcionalidad extra. Es tu oportunidad de demostrar lo que sabes."
